# Core Concepts

Understanding dev-workflow's domain model helps you use the system effectively.

## The Workflow Hierarchy

```
Project
  └── Issue (Feature, Bug, Enhancement, Task)
        └── Plan
              └── Task (with dependencies)
                    └── Execution Session
                          └── PR
```

## Issues

An **Issue** represents a unit of work to be done. Issues are the top-level entity in dev-workflow.

### Issue States

| Status | Description |
|--------|-------------|
| `PLANNED` | Issue created, no plan yet or plan not activated |
| `OPEN` | Plan activated, tasks available for work |
| `IN_PROGRESS` | At least one task is being worked on |
| `CLOSED` | All work complete |

### Issue Types

| Type | Description |
|------|-------------|
| `FEATURE` | New functionality |
| `BUG` | Something broken |
| `ENHANCEMENT` | Improvement to existing functionality |
| `TASK` | Generic work item |

You can also create custom types for your project.

### Issue Properties

- **Title**: Brief description
- **Description**: Detailed explanation
- **Type**: FEATURE, BUG, ENHANCEMENT, TASK, or custom
- **Priority**: LOW, MEDIUM, HIGH, CRITICAL
- **Acceptance Criteria**: Checklist for completion
- **Labels**: Custom key-value tags
- **Milestone**: Optional time-bounded grouping

## Plans

A **Plan** breaks down an issue into actionable tasks. Plans are generated by Claude based on the issue requirements.

### Plan Properties

- **Summary**: Brief description of the approach
- **Approach**: Detailed implementation strategy
- **Estimated Complexity**: LOW, MEDIUM, HIGH, VERY_HIGH
- **Tasks**: Ordered list of work items

### Plan Lifecycle

1. **Generate**: Claude analyzes the issue and creates tasks
2. **Review**: User approves or requests changes
3. **Activate**: Tasks move from PLANNED to BACKLOG
4. **Execute**: Tasks are worked on and completed

## Tasks

A **Task** is a discrete unit of work within a plan. Tasks should be:

- **Deployable**: Each task produces a working, testable change
- **Independent**: Minimize dependencies between tasks
- **Clear**: Well-defined acceptance criteria

### Task States

```
PLANNED → BACKLOG → READY → IN_PROGRESS → PR_REVIEW → COMPLETED
                                    ↓
                               ABANDONED
```

| Status | Description |
|--------|-------------|
| `PLANNED` | Task defined but plan not activated |
| `BACKLOG` | Plan activated, task waiting |
| `READY` | Dependencies met, available for work |
| `IN_PROGRESS` | Actively being worked on |
| `PR_REVIEW` | PR created, awaiting review |
| `COMPLETED` | PR merged, task done |
| `ABANDONED` | Task cancelled or superseded |

### Task Dependencies

Tasks can depend on other tasks:

```
Task 1: Create user model
Task 2: Add authentication middleware (depends on Task 1)
Task 3: Write auth tests (depends on Task 2)
```

A task moves to READY only when all its dependencies are COMPLETED.

### Task Properties

- **Title**: Brief description
- **Description**: What needs to be done
- **Type**: Same types as issues
- **Acceptance Criteria**: Checklist for completion
- **Implementation Plan**: Technical details (for Claude's reference)
- **Labels**: Custom tags
- **Estimated Minutes**: Time estimate

## Execution Sessions

When you start working on a task, Claude creates an **Execution Session**:

### Execution Modes

| Mode | Description | Use Case |
|------|-------------|----------|
| `isolated` | New git worktree | Default, parallel work |
| `branch` | New branch in main worktree | Single task at a time |
| `main` | Work directly on main | Quick fixes, testing |

### Session Lifecycle

1. **Load**: Create worktree/branch, transition to IN_PROGRESS
2. **Work**: Make changes, log progress
3. **PR**: Push branch, create pull request
4. **Review**: Submit for code review
5. **Complete**: Merge PR, cleanup worktree

## Milestones

A **Milestone** groups issues for time-bounded planning:

### Milestone Properties

- **Title**: Name of the milestone
- **Description**: Goals and scope
- **Start Date**: When work begins
- **End Date**: Target completion

### Milestone Status

Status is computed automatically:

| Status | Condition |
|--------|-----------|
| `PLANNED` | No issues have started work |
| `IN_PROGRESS` | At least one issue is being worked on |
| `DELAYED` | Past end date with incomplete work |
| `COMPLETED` | All issues closed (manual sign-off) |

## Snapshots

**Snapshots** provide version history for issues:

- Every significant change creates a snapshot
- View any previous state with time-travel
- Revert to previous versions if needed

### Snapshot Events

- Issue created
- Issue updated
- Plan generated
- Plan regenerated
- Issue closed
- Manual revert

## GitHub Integration

When enabled, dev-workflow syncs with GitHub:

### What Syncs

- **Issues → GitHub Issues**: Local issues create corresponding GitHub issues
- **Tasks → GitHub Issues**: Each task becomes a GitHub issue with labels
- **Status → Project Board**: Task status maps to board columns
- **PR → Task**: PRs link back to their tasks

### Column Mapping

| Task Status | Default Column |
|-------------|----------------|
| BACKLOG | Backlog |
| READY | Ready |
| IN_PROGRESS | In Progress |
| PR_REVIEW | In Review |
| COMPLETED | Done |
| ABANDONED | Done |
